# Deep Research 架构优化方案

## 结论

本项目采用“保留当前多 Agent 编排优势，补齐持久记忆、硬约束、目标函数”的优化路线。

不建议把当前项目重写成另一个 `ascent-research`。更合适的方向是：在保留当前 `orchestrator + scoping + research + verification + report` 分工体系的前提下，吸收 `ascent-research` 在以下几个方面的核心设计：

- 以结构化状态作为系统权威事实，而不是只依赖 Markdown 和 prompt
- 以代码级约束兜底流程纪律，而不是只靠模型“理解并遵守”
- 以可计算的完成标准驱动迭代停止，而不是靠模型主观判断“已经完成”
- 以可追加、可恢复、可跨会话延续的研究记忆替代一次性流水线产物

这个方向既能保留当前项目在多 Agent 分工、人工审批、验证关口、技能扩展、MCP 扩展上的设计亮点，也能把 `ascent-research` 最值得借鉴的底层能力融合进来。

---

## 一、现状判断

### 当前项目的亮点

当前项目已经具备一条相对清晰的研究流水线，几个设计是应该保留的：

- 多 Agent 分工清晰：`scoping`、`research`、`verification`、`report` 职责边界明确
- `scoping` 阶段带人工审批，能有效避免错误方向上的资源浪费
- `verification` 独立成专门环节，而不是把校验混在研究或写作里
- 基于 Deep Agents 的 `StateBackend`、技能加载、MCP 接入，扩展性较好
- OpenAI-compatible 模型接口和多模型架构合理，不应轻易推翻

### 当前项目的主要短板

和 `ascent-research` 相比，当前项目的短板不在“Agent 数量少”，而在基础设施层偏软：

- 研究状态主要依赖 Markdown 文件，缺少真正的结构化权威状态
- 流程纪律主要写在 prompt 里，代码层约束还不够强
- 是否完成、是否继续迭代，缺少统一的机器可判定目标函数
- 研究产物偏单次运行导向，跨会话延续和知识沉淀能力弱
- source 的生命周期、去重、引用一致性、完成度评估都还不够系统化

### `ascent-research` 最值得迁移的精华

本次优化不追求照搬其 CLI 形态或文件布局，而是迁移以下设计原则：

1. 事件账本和结构化状态才是系统真相
2. Agent 输出必须通过类型化契约进入系统
3. 关键规则必须由代码强制执行
4. 系统停止条件必须由 objective function 驱动
5. 研究成果必须支持追加、恢复、复用，而不是每次从头来

---

## 二、目标架构

目标不是把本项目变成“单循环研究引擎”，而是把它升级成“带持久记忆和硬约束的多 Agent Deep Research 系统”。

### 架构方向

在保留现有多 Agent 协作框架的前提下，将系统演进为三层结构：

1. **编排层**
   继续由当前 Orchestrator 驱动整体流程，负责计划、拆解、派发、回收、决策。

2. **状态与约束层**
   新增结构化研究账本、覆盖率评估器、流程守卫、错误码和状态机，使系统行为不再主要依赖 prompt 约束。

3. **记忆与知识层**
   在现有线程态工作区之上增加跨会话可持续的研究记忆，用于保存来源、主题知识、历史结论和可复用研究资产。

### 目标状态

完成优化后，系统应逐步具备以下能力：

- 一次研究运行中的关键状态可以结构化追踪、校验和恢复
- 每个 Agent 的核心输出都有对应的机器可读版本
- Orchestrator 基于 coverage / readiness 决定是否继续迭代，而不是靠主观感觉
- 研究 source 从发现到引用的整个生命周期可追踪
- 同一主题的研究可以跨线程、跨时间延续，而不是重复劳动
- 验证从“纯文字审稿”升级为“文字说明 + 结构化判定”

## 三、总体设计原则

### 1. 保留当前项目的“多 Agent 流水线”主体

当前项目最有价值的地方之一，就是把深度研究拆成了可解释的角色分工。这种结构天然适合复杂问题拆解、并行研究和阶段性质量控制，不能因为学习 `ascent-research` 就把它推倒重来。

因此，后续所有优化都以“增强现有编排体系”为前提，而不是改造成另一套单 Agent 或单循环体系。

### 2. Markdown 继续保留，但不再承担权威状态职责

Markdown 适合人读、人改、人审阅，应该继续保留。

但系统内部决策不能主要依赖 `research_brief.md`、`research_verification.md`、`final_report.md` 这种自由文本。后续需要为关键节点增加对应的结构化产物，例如 JSON 或 JSONL，把 Markdown 降级为“展示层”和“协作层”。

### 3. prompt 负责引导，代码负责兜底

流程规范、角色约束、写作要求仍然可以写在 prompt 里，但关键纪律不能只放在 prompt 里，例如：

- brief 未批准不能启动研究
- verification 未通过不能进入 report
- 研究迭代只能补 gap，不能整轮重来
- 搜索预算耗尽后必须收敛

这些都应由执行逻辑在代码层强制保证。

### 4. 完成标准必须可计算

系统不应主要依赖“verification-agent 觉得可以”或“report-agent 说已完成”来收尾。后续需要让系统对“是否完成”形成统一判定接口，例如 coverage / readiness evaluator。

### 5. 知识默认追加，不默认覆盖

`ascent-research` 很强的一点，是它把研究看成不断积累的过程，而不是一次性产出。

当前项目后续在做持久记忆时，也应默认采用追加式沉淀策略，避免每次运行都把已有结论、来源和知识页推翻重写。

## 四、分阶段优化计划

以下阶段是推荐实施顺序。每个阶段先建立新的基础能力，再进入下一阶段，避免“表层功能很多，底层却不稳”。

### Phase 1：建立结构化研究账本，补齐权威状态层

#### 目标

先把单次研究运行的状态做实，让系统第一次拥有真正的结构化权威状态。

#### 本阶段核心动作

- 在现有 Markdown 产物之外，引入一组结构化研究文件
- 为研究运行建立统一的运行上下文和状态快照
- 为 source 建立明确的生命周期记录
- 为验证结果建立机器可读表示

#### 建议新增的结构化产物

- `research_run.json`
  记录本次研究的基础信息、阶段状态、轮次、预算、时间戳等
- `research_events.jsonl`
  记录本次研究过程中的关键事件，作为轻量事件账本
- `research_brief.json`
  对应 `research_brief.md` 的结构化版本
- `sources.json`
  统一保存 source 元数据、来源类型、状态、去重标识、引用关系
- `verification.json`
  对应 `research_verification.md` 的结构化版本

#### 阶段收益

- 系统状态不再散落在若干自然语言文件里
- 后续的 coverage、恢复、跨会话记忆都有了统一落点
- 后续测试可以直接断言结构化结果，而不是解析长文本

#### 本阶段边界

- 暂不做跨会话长期记忆
- 暂不重构报告渲染层
- 暂不扩展更多输入通道

---

### Phase 2：把流程纪律下沉到代码，建立硬约束执行层

#### 目标

把当前写在 prompt 里的关键规则，迁移成系统级硬约束。

#### 本阶段核心动作

- 在 Orchestrator 执行路径中加入明确的阶段守卫
- 为关键拒绝场景定义统一的机器可读错误或状态码
- 为各个 Agent 的关键输出建立结构化 contract
- 限制 Agent 只能在允许的阶段产出对应结果

#### 重点约束

- `scoping` 未批准，不允许进入研究阶段
- `research` 结果若未满足最基本结构要求，不允许进入验证阶段
- `verification` 若给出关键缺口，不允许直接生成最终报告
- 研究迭代只能围绕缺口补充，而不是再次全局搜索
- 超出预算后必须进入收敛逻辑，而不是继续扩搜

#### 阶段收益

- 流程更稳，行为更可预测
- 减少因为模型误解 prompt 导致的跑偏
- 系统从“prompt 驱动型工程”迈向“规则驱动型工程”

#### 本阶段边界

- 不要求一次把所有软规则都下沉
- 只优先处理真正影响结果质量和成本控制的规则

---

### Phase 3：建立 coverage / readiness evaluator，形成客观完成标准

#### 目标

让系统具备统一的“完成判定器”，把研究是否继续、是否终止、是否允许写报告，交给机器可计算的目标函数。

#### 本阶段核心动作

- 新增 coverage / readiness 评估模块
- 明确定义哪些指标会阻止进入最终报告阶段
- 将 Orchestrator 的收敛决策改为基于 evaluator 输出
- 让 verification-agent 的文本报告与 evaluator 的结构化结果互相对齐

#### 建议优先纳入的指标

- 子问题覆盖率是否达标
- 每个子问题是否有足够证据支撑
- 是否存在被引用但从未抓取的来源
- 是否存在抓取后未被使用的来源
- 是否存在重复研究和明显的任务重叠
- 是否存在高优先级 gap 尚未关闭
- 是否达到本次研究的最小质量门槛

#### 阶段收益

- 系统可以更稳定地判断“还要不要继续”
- verification 不再只是意见，而是参与系统状态转移
- 后续做自动评测、回归测试会简单很多

#### 本阶段边界

- 先聚焦文本研究产物，不强求图表、HTML、附录等复杂指标
- 先保证 evaluator 可用，再逐步加严

---

### Phase 4：从线程工作区升级到跨会话研究记忆

#### 目标

在不破坏当前 `StateBackend` 工作模式的前提下，引入跨线程、跨会话的持久研究记忆。

#### 核心思路

采用“双层存储”：

- 线程内工作区继续使用 `StateBackend`，负责本次运行的临时产物
- 新增持久存储区，负责保存可复用研究资产

这一步不建议照搬 `ascent-research` 的磁盘 session 目录结构，而是更贴合当前 Deep Agents 体系，使用“工作区 + 持久区”的模型。

#### 本阶段核心动作

- 引入 `CompositeBackend` 或等价的持久路由机制
- 为来源、主题、实体、研究结论建立长期保存路径
- 设计 append-first 的知识沉淀策略
- 建立从本次研究结果向长期记忆回写的机制

#### 建议沉淀的长期资产

- 研究主题摘要
- 已接受来源及其元数据
- 可复用的研究 brief 模板和结论摘要
- 面向主题或实体的知识页
- 历史研究运行的简要索引

#### 阶段收益

- 系统第一次具备“研究延续性”
- 同类问题复查时，不必每轮都从头开始
- 后续可以逐步长出 wiki、主题页、研究档案等能力

#### 本阶段边界

- 先做持久记忆，不急着追求复杂知识图谱
- 先保证写入和读取语义稳定，再考虑高级检索和引用

---

### Phase 5：扩展输入路由与输出形态，提升系统完整度

#### 目标

在状态层、约束层、评估层、记忆层稳定后，再扩展系统边界能力。

#### 输入侧优化方向

- 建立 source routing 机制，区分 Web 搜索、MCP 数据源、可选本地文件、未来可选 browser 抓取
- 为抓取增加基础质量门槛、重复检测、来源分层和失败分类
- 逐步摆脱“所有研究都等于 Tavily + 网页正文抓取”的单一路径

#### 输出侧优化方向

- 从最终 Markdown 报告逐步扩展到更强的渲染形式
- 支持更丰富的来源附录、研究摘要卡、结构化导出
- 在未来合适的时候，再考虑图表、HTML 报告、双语输出等增强能力

#### 为什么把这一步放最后

因为这些能力都是“放大器”，不是“稳定器”。

如果底层没有研究账本、coverage 和持久记忆，这些增强能力只会把系统复杂度做高，不会显著提升结果质量。

---

## 五、需要刻意保留的当前设计亮点

后续优化过程中，以下特性应明确视为“保留项”，不能在重构时被误伤：

### 1. 保留多 Agent 分工，不退化成单 Agent 大杂烩

当前的多角色分工让研究任务更容易解释、调试和局部优化，这是本项目的重要资产。

### 2. 保留人工审批在 scoping 阶段的前置作用

这是当前项目在成本控制和方向控制上的关键护栏，应继续保留，未来只是把审批结果结构化，而不是取消。

### 3. 保留 verification 作为独立关口

verification 不应该并回 research 或 report，而应继续作为独立角色存在，只是后续要和结构化 evaluator 协同工作。

### 4. 保留技能体系和 MCP 扩展能力

这些能力决定了项目的领域适配潜力，应该在新的状态层之上继续工作，而不是被架构调整打断。

### 5. 保留 OpenAI-compatible、多模型、低温度配置

这些配置符合当前项目的定位和使用方式，不需要因为学习其他项目而轻易改变。

---

## 六、不建议当前阶段做的事

为了保证演进节奏正确，以下事项不建议提前启动：

- 不建议直接把项目整体改造成 `ascent-research` 风格的单循环系统
- 不建议优先做图表、SVG、HTML 报告等表现层能力
- 不建议一开始就引入复杂知识图谱或过度设计的 wiki 系统
- 不建议在没有 coverage evaluator 之前就做过多自动迭代策略
- 不建议把所有规则都继续堆在 prompt 里，而不下沉代码

---

## 七、预期结果

当以上路线按阶段推进后，本项目会从当前的“多 Agent 研究流水线”演进为：

一个既保留现有编排优势，又具备结构化状态、代码级流程守卫、客观完成标准和跨会话研究记忆的 Deep Research 系统。

它不会失去当前项目在角色分工和扩展性上的优点，也不会简单模仿 `ascent-research` 的外在形态，而是把其最有价值的底层设计原则，转化为适合当前 Python + Deep Agents + LangGraph 技术栈的系统能力。

这也是当前项目下一阶段最合理、最稳妥、长期收益最高的演进方向。
